# CLAUDE.md — дизайн-бриф проєкту (CRM: телефони, месенджери, ціни, соцмережі)

> Цей файл — персистентні інструкції для Claude Design у цьому проєкті. Читається один раз на корені проєкту й діє на всі наступні генерації, поки не зміниш.

## Роль

Ти — дизайн-лід продуктової студії, що спеціалізується на внутрішніх бізнес-інструментах для операційних команд із високим навантаженням — рівень Linear, Retool, Vercel dashboard. Клієнти приходять сюди саме тому, що втомились від дефолтних адмінок, які всі виглядають однаково.

Продукт: внутрішня CRM для менеджерів і адміністраторів, які щодня керують мережею сайтів і піддоменів — номери телефонів, месенджер-акаунти, ціни, соціальні мережі, адреси. Це не публічний продукт і не маркетинговий сайт. Ним користуються по 8 годин на день. Якість = швидкість сканування очима, однозначність статусів, мінімум кліків. Не "вау"-ефект.

Мова інтерфейсу за замовчуванням — українська, якщо в конкретному промті не вказано інше.

## Естетичний напрям — явно проти дефолту

Без прямої вказівки ти тяжієш до двох однаково поганих крайнощів:
1. **"Маркетинговий AI-шаблон"** — Inter/Roboto, фіолетові градієнти, картки по три, зайва анімація.
2. **"Дефолтна адмінка"** — сіро-білі картки, синій Bootstrap-акцент, однакові рамки на всьому, бейджі випадкових кольорів без системи, типовий sidebar+topbar без жодної індивідуальності.

Обидва — шаблон, не рішення. Візьми одну точку зору й витримай її по всій системі. Якщо в контексті є skill Frontend Design — виклич і використай його для типографіки й кольору; нижче — обмеження поверх нього, специфічні для CRM.

Напрям-приклад (можеш вдосконалити, тримайся духу):
- Основа: темна/графітова, не чорна і не сіро-біла за замовчуванням.
- Один теплий/сигнальний акцент — і тільки він означає увагу/статус/дію. Жодних випадкових кольорів бейджів "на око".
- Типографіка: дисплей-шрифт із характером для заголовків розділів (не Inter), шрифт із табличними цифрами для даних і таблиць, моно — для токенів, номерів, кодів синхронізації.
- Щільність: погана CRM — тісна; хороша CRM — щільна, але зрозуміла. Ціль — друге, не менше даних на екрані заради "повітря".

**Заборонено:** Inter як єдиний шрифт, фіолетові градієнти, дефолтні тіні Tailwind, емодзі замість іконок, кольорові бейджі без легенди, декоративні ілюстрації, маркетингові hero-секції.

## Обов'язкова інформаційна архітектура

### 1. Сайти та ієрархія
Додавання сайту; додавання піддомену — піддомен завжди закріплений за основним сайтом (вкладена структура, видно в списку й у breadcrumbs). Групи: створення групи, зв'язок сайт↔група (сайт може входити в кілька груп), фільтр списку по групі. Обране — позначка на сайті, окремий швидкий доступ.

### 2. Пошук і фільтри
Комбіновані фільтри одночасно (не один dropdown): сайт/піддомен, група, статус, тип даних, гео. Пошук охоплює й вкладені піддомени.

### 3. Історія змін
Журнал на трьох рівнях: глобальний (усі сайти), по сайту, по типу даних (телефони / месенджери / ціни / соцмережі / адреси). Кожен запис: хто, коли, що саме, було → стало.

### 4. Масові дії
Окремо для кожного типу даних (телефони, месенджери, ціни, соцмережі): вибір кількох рядків → пакетна зміна значення, зміна гео-правила, видалення. Перед підтвердженням — явний прев'ю "що саме зміниться в N елементах".

### 5. Дані сайту й гео-правила на кожен слот
Кожен сайт має власний набір слотів даних. Кожен слот (наприклад, номер телефону) має гео-режим:
- показувати всім;
- показувати всім, **крім** обраних країн (виключення);
- показувати **тільки** обраним країнам.

Кожен режим — свій впізнаваний бейдж/іконка, видима одразу в таблиці без відкриття картки (напр.: глобус = всім, глобус із закресленим прапорцем = крім обраних, прапорець у рамці = тільки обраним). Іконографія однакова в усій системі.

Адреси можуть прив'язуватись до телефону (один номер → кілька адрес).

### 6. Телефони — маска відображення
Номер завжди зберігається у єдиному канонічному вигляді (формат на кшталт E.164: +[код][номер], без пробілів і символів). Окремо, на рівні сайту чи слота — конструктор ФОРМАТУ ВІДОБРАЖЕННЯ: шаблон типу `+# (##) ###-##-##`, `+## ##-###-####` тощо, з живим прев'ю "як це виглядатиме на сайті" одразу біля поля вводу маски.

### 7. Резерви номерів (закласти місце вже зараз)
Спроєктуй слот телефону так, щоб він міг показати ланцюжок: активний номер → резерв 1 → резерв 2 → значення за замовчуванням / приховати. У таблиці — статус кожного номера в ланцюжку (активний / резерв / недоступний), навіть якщо зараз активний лише один номер. Інтерфейс не має ламатись, коли резервів стане кілька.

### 8. Права доступу
Матриця: менеджери × сайти, клітинка — рівень доступу (чекбокс або перемикач). Ролі: адміністратор (усе) і менеджер (обмежено матрицею).

### 9. Токени та синхронізація
Створення/відкликання токена (API-ключа) на сайт. Кнопка "Синхронізувати" зі статусом: успіх / помилка / час останньої синхронізації.

### 10. Редагування даних
Обери один патерн — інлайн у таблиці **або** бокова панель — і застосуй послідовно для всіх типів даних. Явні стани: збережено, помилка, є незбережені зміни.

### 11. Авторизація
Логін, ролі, вихід, сесії — стандартно, без експериментів. Це не місце для творчості.

## Як показувати статуси й дані

Вирівняні таблиці для списків (сайти, телефони, ціни) — не розсипай у карткові гріди там, де таблиця читається швидше. Один узгоджений набір кольорів/іконок для станів (активний / неактивний / резерв / помилка / очікує), із видимою в інтерфейсі легендою. Порожні, помилкові й завантажувальні стани — теж спроєктовані, не пропущені.

## Тон текстів в інтерфейсі
Називай речі так, як їх бачить менеджер, а не як влаштована система (не "webhook config", а "налаштування синхронізації"). Дії — активним станом ("Зберегти", не "Підтвердити"). Помилки — конкретні, без вибачень і без розпливчастості.

## Порядок роботи перед генерацією екрана
1. Зафіксуй компактну систему токенів (колір / тип / відступ / іконографія станів) — одну на весь проєкт.
2. Звір із цим брифом: якщо щось виглядає як типовий дефолт для будь-якої CRM, а не рішення саме для цієї задачі — перероби.
3. Тільки тоді генеруй екран.

Мінімальний набір екранів для першого проходу: список сайтів (групи / обране / пошук), деталі сайту (піддомени + усі типи даних), один тип даних детально (телефони — гео-бейджі, маска, резерви), модалка масової дії, матриця доступу, журнал змін.

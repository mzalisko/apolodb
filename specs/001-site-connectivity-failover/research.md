# Research - Feature 001: Site registration & connection-status monitoring

**Phase 0 output** (`/speckit.plan`). Source: workflow `plan-001-connectivity` (research agents: CRM backend / WP plugin / proxy-security), verified against constitution + spec-consistency.

## CRM Backend (Laravel)

### Laravel version

- **Decision**: Laravel 13.x (current stable major, PHP 8.3–8.5) — treat as the constitution's «стабільна LTS-версія».
- **Rationale**: Конституція (Принцип V) вимагає «стабільну LTS-версію Laravel» та PHP >= 8.3. Станом на 2026-07-21 сучасний Laravel більше НЕ маркує релізи як «LTS» — з Laravel 6 усі мажорні релізи мають однакову політику підтримки: виправлення багів 18 місяців, безпекові виправлення 2 роки. Тому «LTS» тут читаємо як «поточний стабільний мажор із найдовшим вікном активної підтримки». За даними офіційної support policy (context7 /laravel/docs): Laravel 13 випущено 2026-03-17, вимагає PHP 8.3–8.5, багфікси до ~Q3 2027, безпека до 2028-03-17. Це і найдовше вікно, і єдиний мажор, чия МІНІМАЛЬНА вимога PHP (8.3) точно збігається з конституційним порогом PHP 8.3 — тобто на всьому підтримуваному діапазоні PHP немає застарілих версій нижче 8.3. Laravel 12 (реліз 2025-02, PHP 8.2–8.5) уже входить у фінальну фазу — багфікси спливають 2026-08-13, тобто за ~3 тижні від дати планування, тож стартувати новий проєкт на 12 недоцільно.
- **Alternatives considered**: Laravel 12 — відхилено: вікно багфіксів спливає 2026-08-13 (майже одразу), мінімальний PHP 8.2 нижчий за конституційний поріг. Laravel 11 — відхилено: багфікси вже завершені (2025-09), лишається тільки безпека до 2026-09. Очікування на Laravel 14 — відхилено: не випущений, немає сенсу блокувати старт. Пін на конкретний патч — не потрібен, тримаємо ^13.0 через Composer constraint.

### Database engine

- **Decision**: PostgreSQL 16+ для CRM-бекенду (окремо від MySQL, який використовує WordPress-плагін на своєму боці).
- **Rationale**: Робоче навантаження — це часті ідемпотентні upsert'и поточного статусу (latest-wins) від ~500 сайтів + періодичний set-based UPDATE детектора офлайну + аудит-лог. Обсяг малий (8–9 записів/с), тож будь-яка СУБД впорається, але вирішальні якісні фактори схиляють до PostgreSQL: (1) MVCC дає неблокуючу конкурентність між безперервними upsert-heartbeat'ами та періодичним sweep'ом офлайну без ескалації блокувань; (2) чистий `INSERT ... ON CONFLICT ... DO UPDATE` для ідемпотентного latest-wins (Laravel `upsert()`); (3) часткові індекси (`WHERE status='online'`) роблять і детектор офлайну, і фільтр списку «N із M» дешевими; (4) сильна цілісність (справжні CHECK-констрейнти, суворі типи, транзакційний DDL) для моделі довіри конституції; (5) джерело істини автентифікації та per-site секретів (FR-029) виграє від строгих гарантій. CRM-БД навмисно відокремлена від MySQL плагіна — це узгоджується з Принципом I (жодного взаємопроникнення).
- **Alternatives considered**: MySQL 8 / MariaDB — повністю прийнятна альтернатива, добре підтримується Laravel і звична WP-екосистемі; відхилено як основну лише через слабші часткові індекси та менш чисту семантику конкурентних upsert/sweep. Якщо команда сильніша в MySQL — це не блокер, ре-рішення допустиме на етапі плану. SQLite — лише для локальних тестів, не для 500+ сайтів у проді.

### Queue driver

- **Decision**: Redis як черга (+ Laravel Horizon для моніторингу воркерів); БД-черга допустима лише як тимчасовий MVP-фолбек.
- **Rationale**: Принцип IV вимагає асинхронної обробки рутинних heartbeat'ів. Навантаження ~8–9 повідомлень/с усталено, але з бурстами до 500 одночасних звітів (Edge Case «пік 500+», SC-007). БД-черга під таким профілем страждає від polling-затримки та конкуренції за блокування рядків `FOR UPDATE SKIP LOCKED`, надаючи додаткове навантаження на ту саму БД, що обслуговує upsert'и. Redis дає низьколатентний push/pop, чистіше горизонтальне масштабування воркерів і рівномірне згладжування бурстів; Horizon додає видимість глибини черги/throughput (корисно для верифікації SC-007). Той самий Redis-інстанс паралельно закриває суміжні потреби: кеш і незалежний per-site rate-limiting бекенду (другий контур FR-026). Обробка heartbeat: контролер приймання виконує лише дешеву валідацію + авторитетну HMAC/timestamp-перевірку (FR-028), ставить `ProcessHeartbeat` job у чергу і одразу повертає 202/204; job виконує ідемпотентний upsert статусу та last_seen_at.
- **Alternatives considered**: Database queue — відхилено як основну через polling-затримку, lock-contention і додаткове навантаження на робочу БД під бурстами; лишається валідним для найранішого MVP, якщо Redis ще не піднято. SQS/інші хмарні черги — зайва зовнішня залежність (презумпція відмови, Принцип V). Синхронна обробка — прямо заборонена Принципом IV.

### Heartbeat storage model

- **Decision**: Latest-wins: поточний статус + last_seen_at зберігаються ідемпотентним upsert'ом (окрема таблиця site_statuses 1:1 із sites або колонки на sites). Кожен heartbeat НЕ зберігається. Ведеться лише обмежена історія ЗМІН статусу (транзицій) для аудиту/чейнджлогу.
- **Rationale**: Спека вимагає лише поточний статус + час останнього оновлення (FR-012, FR-015, FR-017; сутність «Звіт про статус» обробляється чергою, а не архівується). Ідемпотентний upsert по унікальному site_id гарантує, що повтори/гонки двох воркерів дають узгоджений latest-wins без дублів (Edge Case «двоє операторів», «пік навантаження»). Зберігати КОЖЕН heartbeat недоцільно: 500 сайтів × 1/хв = ~720k рядків/добу без вимоги спеки — марне роздування БД. Натомість пишемо рядок в audit/changelog ЛИШЕ при зміні статусу (online→offline, offline→online, revoke тощо) — це і задовольняє FR-021 та `design/docs/CRM/09-changelog.md`, і тримає історію обмеженою й корисною. Поле last_used_at секрету (сутність Token) оновлюється в тому ж upsert.
- **Alternatives considered**: Повний журнал кожного heartbeat (append-only) — відхилено: спека не вимагає, необмежене зростання, потребувало б партиціонування/prune без цінності. Тільки колонки на sites без окремої таблиці — прийнятно для MVP, але 1:1 site_statuses ізолює «гарячі» write-колонки від рідко змінюваних реєстраційних даних сайту (менше write-amplification на індексах sites). Зберігати історію в тій самій таблиці статусу — відхилено: змішує latest-wins із append-only.

### Offline detection mechanism

- **Decision**: Планована консольна команда `sites:detect-offline` раз на хвилину через Laravel Scheduler (withoutOverlapping), один set-based UPDATE, що переводить у «офлайн» усі сайти зі status='online' та last_seen_at < now() − offline_window (дефолт 3 пропущені звіти ≈ 5 хв). CRM НЕ пінгує сайти.
- **Rationale**: FR-014 прямо забороняє пінг серверів з боку CRM — офлайн визначається виключно відсутністю push-звітів (last_seen_at). Детекція = одне індексоване UPDATE-твердження (частковий індекс `(status, last_seen_at) WHERE status='online'`), а не per-row цикл; на масштабі 500 сайтів це мілісекунди й тривіально масштабується навіть до десятків тисяч. Запуск щохвилини планувальником Laravel (`schedule->command(...)->everyMinute()->withoutOverlapping()`), що узгоджується з інтервалом звітування ~1 хв і вікном ~5 хв (A-2, SC-003). Вікно офлайну — конфігуроване (config/env), бо спека називає значення «базовими дефолтами». Кожна транзиція online→offline емітує подію для запису в changelog. Для >500 масштабу нічого не змінюється — UPDATE лишається set-based; єдина операційна вимога — стабільний cron/`schedule:run` (або systemd timer) на одному вузлі, `withoutOverlapping` захищає від накладання прогонів. Симетрично, upsert heartbeat повертає сайт з offline у online, тож окремого «повернення» команда не потребує.
- **Alternatives considered**: Per-site заплановані завдання/затримані jobs на кожен сайт (напр. dispatch job з delay = offline_window) — відхилено: 500+ окремих таймерів, складніше, зайве навантаження на чергу проти одного UPDATE. Обчислювати офлайн «на льоту» при рендері списку без матеріалізації статусу — відхилено: FR-013/FR-016 вимагають визначеного збереженого статусу для фільтра «N із M» та подій змін; ліниве обчислення не дає точки емісії аудит-події транзиції. Активний пінг з CRM — прямо заборонено FR-014.

### Testing stack & critical-path coverage

- **Decision**: Pest (поверх PHPUnit) як тест-фреймворк за замовчуванням; PHPUnit-сумісність зберігається. Автотести обов'язкові для критичних шляхів, визначених quality gates конституції, перед злиттям.
- **Rationale**: Skeleton Laravel 12/13 постачається з Pest як типовим вибором; він дає лаконічніший синтаксис поверх PHPUnit без втрати сумісності, тож жодної нової важкої залежності. Конституція («Робочий процес і контроль якості») прямо називає критичні шляхи, що ПОВИНЕН мати автотести до злиття: (1) автентифікація — валідація HMAC-підпису + вікна timestamp, відхилення незаавтентифікованих/невалідних/відкликаних запитів (FR-007, FR-011, SC-004); (2) захист від replay — відхилення повторів поза вікном толерантності (FR-011, SC-005); (3) per-site ізоляція даних — сайт бачить лише власні дані, відповіді не розкривають інші сайти (FR-019, FR-032); (4) повторна незалежна верифікація на бекенді (defense-in-depth) — бекенд не довіряє проксі-заголовкам наосліп (FR-028); (5) відкликання/ротація токена набуває чинності майже одразу (FR-005, SC-006); (6) ідемпотентність heartbeat-upsert та latest-wins під конкурентністю; (7) команда детекції офлайну (перехід у офлайн рівно після вікна, час від last_seen_at — SC-003); (8) заборона дублю домену (FR-006). Тип: feature-тести для ендпоінта приймання та екрана списку, unit-тести для HMAC-підписувача й логіки вікна часу.
- **Alternatives considered**: Чистий PHPUnit — відхилено як дефолт лише стилістично (Pest над ним і сумісний), цілком допустимий, якщо команда надає перевагу класичному синтаксису. Мінімальне покриття «щасливого шляху» без безпекових/ізоляційних тестів — відхилено: прямо порушує quality gate конституції (безпекові порушення блокують злиття без винятків).

### Frontend for site-list screen

- **Decision**: Blade-шаблони + Vite + npm; усі шрифти/CSS/JS зібрані й збережені локально в репозиторії, без жодного CDN. Alpine.js (через npm, у бандлі) — опційно для інтерактивності фільтра; Tailwind локально.
- **Rationale**: Принцип III вимагає нуль зовнішніх runtime-залежностей на фронтенді: жодних зовнішніх <script>/<link>, жодного CDN для шрифтів/стилів/скриптів; збірка через npm + Vite з локальним зберіганням. Принцип V робить Blade шаблонізатором за замовчуванням і дозволяє Alpine.js/Livewire ЛИШЕ якщо вони не тягнуть зовнішніх runtime-залежностей. Екран списку сайтів (User Story 3, `design/docs/CRM/02-sites-list.md`) — це серверно-рендерена Blade-сторінка: рядок = сайт/домен + поточний статус за єдиною легендою кольорів (онлайн→зелений, очікує→синій, офлайн→червоний, неактивний→сірий; FR-016) + час останнього оновлення (A-7), плюс фільтр за статусом із лічильником «N із M» (FR-018). Оновлення не мусить бути real-time (FR-017), тож достатньо перезавантаження/легкого Alpine-фільтра — WebSocket не потрібен. Шрифти встановлюються npm-пакетом і бандляться Vite (self-hosted), щоб з адмінки не було жодного зовнішнього звернення. CI-гейт має відхиляти будь-які зовнішні CDN-посилання (quality gate Принципу III).
- **Alternatives considered**: Livewire — дозволено конституцією, але надлишково для здебільшого статичного списку з не-real-time оновленням; Alpine достатньо і легше. SPA (Vue/React/Inertia) — відхилено: важча залежність без потреби, суперечить вузькому передбачуваному стеку (Принцип V). CDN для шрифтів/Tailwind (напр. Google Fonts, cdn.tailwindcss.com) — прямо заборонено Принципом III; усе self-hosted через Vite. Полінг real-time/WebSocket — не потрібен (FR-017), відхилено як зайву складність.

**Open notes (deferred to plan / implementation):** Прокляті/суміжні рішення, свідомо ВИНЕСЕНІ за межі Phase-0 CRM-бекенду цього запиту (будуть уточнені на етапі плану): (1) Конкретний продукт проксі/ingress-шлюзу (reverse-proxy / API-gateway / CDN з edge-логікою) — FR-023..FR-033, OQ-4 прямо лишає вибір вендора на етап плану; тут зафіксовано лише, що бекенд авторитетно й незалежно верифікує HMAC/timestamp (FR-028) і per-site секрети НЕ виходять на проксі (FR-027/FR-029). (2) Точні криптопараметри HMAC — алгоритм (напр. HMAC-SHA256), ширина вікна толерантності timestamp, анти-replay nonce/кеш — A-4 лишає їх плану; для реалізації знадобиться короткоживучий store для nonce (природно — той самий Redis). (3) Спосіб зберігання per-site секрету на бекенді: секрет показується один раз (FR-004), далі зберігається так, щоб не читатись у відкриту — на етапі плану уточнити хеш/шифрування-at-rest та формат публічного site-ідентифікатора. (4) Точні дефолти інтервалів (1 хв звіт / 5 хв офлайн) конфігуровані — закріпити в config/env на етапі плану. (5) Розгортання планувальника: потрібен надійний cron/systemd-timer, що виконує `schedule:run` щохвилини на одному вузлі (з withoutOverlapping у команді детекції офлайну) — деталь інфраструктури деплою, не архітектури БД. (6) Вибір БД (PostgreSQL) і черги (Redis) — рекомендації; якщо команда сильніша в MySQL/database-queue, ре-рішення допустиме на етапі плану без порушення конституції, з відповідним записом у Complexity Tracking плану. Матриця доступу менеджерів (A-5), піддомени (A-6) і вся площина контактних даних/резервних НОМЕРІВ (A-1) — поза межами цієї фічі.

## WordPress Plugin (vanilla PHP)

### Планувальник відправки серцебиття (~1/хв)

- **Decision**: Основний механізм — WP-Cron із зареєстрованим кастомним інтервалом ~60 с (add_filter('cron_schedules') → 'db_every_minute'; wp_schedule_event). Обробник хука надсилає підписаний heartbeat через wp_remote_post(). Оскільки WP-Cron спрацьовує лише за наявності трафіку, у налаштуваннях плагіна документуємо і рекомендуємо для 'тихих' сайтів вимкнути псевдо-крон (define('DISABLE_WP_CRON', true)) і викликати wp-cron.php реальним системним cron (crontab, кожну хвилину, wget/curl або WP-CLI 'wp cron event run --due-now'). Плагін лишається без зовнішнього планувальника-залежності: він тільки реєструє подію, спосіб її запуску — вибір адміністратора сайту.
- **Rationale**: WP-Cron — стандартний, портативний механізм WordPress Plugin API, не тягне зовнішніх залежностей і працює на будь-якому шаред-хостингу (узгоджено з Принципом I: vanilla PHP, WP API, без Laravel; Принцип V: стандарти WP). Він задовольняє вимогу A-2/FR-009 про ~1 звіт/хв за умови регулярного трафіку. Ключовий недолік — WP-Cron не є справжнім cron: він 'ліниво' спрацьовує лише коли хтось відкриває сторінку сайту, тож на низькотрафічному сайті heartbeat запізнюється, що спричинить хибний офлайн (FR-014, SC-003 — офлайн після 3 пропущених). Рішення покриває обидва режими: за замовчуванням WP-Cron (нульове налаштування), а для надійності — документована опція реального системного cron, який б'є wp-cron.php раз на хвилину.
- **Alternatives considered**: (1) Тільки WP-Cron без альтернативи — відкинуто: ненадійно для тихих сайтів, дає хибні офлайни. (2) Власний loopback self-ping (спроба будити крон запитом до себе) — відкинуто: тендітно, дублює логіку ядра WP, б'є по продуктивності. (3) Зовнішній планувальник як обов'язкова залежність плагіна (напр. хмарний cron, що смикає REST-ендпоінт плагіна) — відкинуто: суперечить Принципу II (плагін нічого не знає про зовнішню інфраструктуру, тільки відповідає) і додає компонент. (4) JS-таймер у браузері відвідувача — відкинуто: працює лише при відкритій сторінці, недетерміновано, тягне фронтенд-логіку (Принцип III).

### Зберігання конфігурації (site-id, HMAC-секрет, endpoint URL)

- **Decision**: Використовуємо WordPress Options API з єдиним префіксованим ключем-масивом (напр. get_option('sd_connection')) або трьома окремими опціями: 'sd_site_id' (публічний ідентифікатор), 'sd_signing_secret' (секрет HMAC), 'sd_endpoint_url' (єдина публічна адреса проксі). Секрет: (a) записується через update_option при введенні/ротації; (b) НІКОЛИ не рендериться назад у поле — у UI вкладки 'Подключение' показується лише маскований плейсхолдер ('••••••••' + 'показать/скрыть' діє тільки на щойно введене значення в тій самій сесії форми, не на збережене); (c) при submit порожнього поля секрет не перезаписується; (d) autoload='no' для секрету. Endpoint URL і site-id — несекретні, можуть показуватися повторно. Опційно підтримуємо override через wp-config константу (define('SD_SIGNING_SECRET', ...)), тоді значення береться з env/константи, а не з БД.
- **Rationale**: Options API — канонічне сховище налаштувань плагіна у WP (Принцип V), не потребує власних таблиць і працює скрізь. Розділення на публічний site-id (для пошуку сайту на бекенді, FR-002) і окремий секрет підпису (FR-002/FR-003) відповідає уточненню A-4. Політика 'секрет показується один раз, далі не читається у відкриту' на боці плагіна дзеркалить FR-004/FR-024 і вимогу конституції 'секрети зберігаються захищено, як секрети — на боці плагіна'. Endpoint задається конфігурацією (одне значення = публічний ендпоінт проксі), а не константою в коді — прямо виконує FR-024 та Принцип II (жодних захардкоджених адрес). Підтримка wp-config константи дає шлях тримати секрет поза БД (узгоджено з вимогою 'секрети — через env/сховище', хоча на чужому WP-хостингу це опція, не гарантія).
- **Alternatives considered**: (1) Власна таблиця в БД — відкинуто: надлишково для трьох значень, суперечить легкості плагіна (Принцип I). (2) Зберігати секрет у коді/файлі плагіна — відкинуто прямо: заборонено Принципом II і секцією 'Вимоги безпеки' (секрети не в репозиторії/коді). (3) Ехати секрет назад у HTML value поля для зручності — відкинуто: порушує FR-024 (секрет ніколи не показується повторно) і ризикує витоком через кеш/історію браузера. (4) Шифрувати секрет власним ключем у БД — відкинуто на цьому етапі: ключ шифрування все одно жив би поруч у тому ж WP, дає хибне відчуття безпеки; надійніший шлях — опційна wp-config константа.

### HMAC-підпис запиту у vanilla PHP (канонічний рядок + заголовки)

- **Decision**: Плагін підписує кожен запит через hash_hmac('sha256', $canonical, $secret) і передає підпис у hex. Канонічний рядок — фіксований порядок полів, з'єднаних '\n' (LF), над сирим тілом:
  canonical = site_id + "\n" + timestamp + "\n" + nonce + "\n" + sha256_hex(raw_body)
де timestamp — Unix-секунди (UTC, time()), nonce — унікальне випадкове значення на запит (bin2hex(random_bytes(16))), raw_body — точні байти JSON-тіла, що йдуть у wp_remote_post (хешуємо тіло, а не вкладаємо його цілком у канонік, щоб уникнути проблем кодування). Підпис = hash_hmac('sha256', canonical, secret). Заголовки запиту:
  Content-Type: application/json
  X-SD-Site-Id: <site_id>
  X-SD-Timestamp: <unix_ts>
  X-SD-Nonce: <nonce>
  X-SD-Signature: sha256=<hex_hmac>
Плагін ТІЛЬКИ підписує; порівняння підпису constant-time (hash_equals) виконує бекенд. Плагін не робить жодної верифікації відповіді, окрім читання статус-коду.
- **Rationale**: hash_hmac і random_bytes — вбудовані у PHP 8.3 (Принцип V), без зовнішніх залежностей. Канонічний рядок із суворо фіксованим порядком і роздільником '\n' усуває неоднозначність, а хешування сирого тіла (sha256_hex(raw_body)) робить підпис стійким до розбіжностей серіалізації/кодування між плагіном і бекендом. Включення timestamp і nonce у канонік прив'язує підпис до часу й унікальності запиту, що дає бекенду захист від replay (FR-011, SC-005) і зміни вмісту. Site-id у каноніку не даємо підмінити без інвалідації підпису. Заголовки несуть усе потрібне бекенду й проксі: проксі робить дешеву несекретну edge-перевірку (наявність/формат X-SD-* та вікно timestamp — FR-027) не знаючи секрету, а авторитетну HMAC-перевірку constant-time (hash_equals) робить виключно бекенд (FR-027/FR-028, конституція: per-site секрет і його перевірка лишаються на бекенді). Роль плагіна свідомо мінімальна — 'тільки підписати'.
- **Alternatives considered**: (1) Вкладати повне тіло у канонічний рядок замість його хешу — відкинуто: крихко до кодування/пробілів, роздуває рядок. (2) Порядок полів за алфавітом чи JSON-канонізація ключів — відкинуто: зайва складність, фіксований конкатенований порядок простіший і детермінований. (3) base64 замість hex для підпису — прийнятно, але hex обрано за простоту транспортування в заголовку й читабельність логів. (4) HMAC-SHA1/MD5 — відкинуто: слабші; SHA-256 — сучасний дефолт. (5) Constant-time порівняння на боці плагіна — не застосовно: плагін не верифікує вхідні підписи, тому hash_equals тут не потрібен (зазначено як відповідальність бекенду).

### Мінімальний payload серцебиття для цієї фічі

- **Decision**: Тіло heartbeat — компактний JSON рівно з полями цієї фічі:
  {
    "site_id": "<публічний site-ідентифікатор>",
    "status": "online",
    "timestamp": <unix_ts_секунди_UTC>,
    "nonce": "<hex_16_байт>"
  }
Підпис НЕ кладеться в тіло — він іде заголовком X-SD-Signature (щоб хеш тіла в каноніку не залежав від самого підпису). Поле status для цієї фічі приймає лише 'online' (плагін звітує лише коли живий; 'offline' виводиться бекендом за відсутністю звітів — push-based, FR-013/FR-014, а не самозвітом плагіна). Ніяких контактних даних (телефони, слоти, ціни, гео) у payload немає.
- **Rationale**: Спека свідомо звужена до сполучного каркасу: сутність 'Звіт про статус (Heartbeat)' складається з ідентичності сайту, статусу підключення, позначки часу й підпису — рівно це й кладемо (Key Entities, FR-009). status='online' достатньо, бо офлайн визначається відсутністю звітів на боці CRM (FR-014), а не самозвітом. Мінімалізм payload прямо виконує обсяг фічі й Принцип I (легкий плагін): синхронізація контактних даних (телефони/ланцюг резервних номерів/ціни/гео) явно поза межами (A-1, 'Поза межами'), тому нічого з цього в heartbeat не потрапляє. timestamp+nonce у тілі дублюють заголовки для зручності бекенду, але канонік хешує саме сире тіло, тож вони узгоджені з підписом.
- **Alternatives considered**: (1) Додати версію плагіна / кількість отриманих слотів / стан останньої синхронізації у heartbeat — відкинуто: це діагностика й дані контактної синхронізації, поза обсягом цієї фічі (A-1). (2) Класти підпис у тіло — відкинуто: створює циклічну залежність (хеш тіла включав би підпис), тому підпис лише в заголовку. (3) Слати і 'online', і явний 'offline' від плагіна — відкинуто: офлайн push-моделлю виявляє CRM за тишею (FR-014), плагін фізично не може надіслати 'я офлайн', коли він офлайн. (4) Ставити ISO-8601 час замість Unix-секунд — прийнятно, але Unix-секунди простіші для перевірки вікна толерантності й уникають розбіжностей таймзон.

### Нейтральна ідентичність, read-only та відповідність стандартам WP

- **Decision**: Плагін тримає нейтральну ідентичність: пункт меню 'Данные сайта' зі стандартною WP-іконкою, без лого/назви DataBridge/CRM, шорткоди з префіксом sd_*, у текстах і в коді жодної згадки централізованої панелі. Жодної захардкодженої внутрішньої адреси — плагін звертається виключно за налаштованим endpoint URL (публічний проксі), і навіть у помилках/тостах ('Проверить соединение') не розкриває топологію. Плагін read-only щодо контактних даних (лише приймає й показує), а для цієї фічі — лише надсилає підписаний статус і читає код відповіді. Код відповідає WP Coding Standards: префіксація всіх функцій/опцій/хуків (sd_/SD_), санітизація вводу (sanitize_text_field, esc_url_raw для endpoint) та ескейпінг виводу (esc_html/esc_attr/esc_url), нонси на формах адмінки, wp_remote_post замість raw cURL, i18n через __()/esc_html__ (RU-аудиторія вебмайстрів), без прямих SQL і без зовнішніх runtime-ресурсів на фронті (Принцип III).
- **Rationale**: Нейтральна ідентичність прямо задана дизайн-доком 01-plugin-overview (меню без брендингу, sd_*, без згадок панелі) і Принципом II: скомпрометований/реверснутий плагін має давати нуль інформації про CRM та інші сайти (FR-019, FR-008/FR-024, FR-032). Відсутність захардкоджених адрес і конфігурований endpoint — це і Принцип II, і FR-008/FR-024. Read-only статус плагіна відповідає ролі 'приймач даних лише для читання' зі спеки та дизайну. Відповідність WP Coding Standards і WP Plugin API — вимога Принципу V і 'Робочого процесу' (зміни в plugin/ перевіряються на відсутність Laravel і захардкоджених адрес CRM); санітизація/ескейпінг/нонси/wp_remote_post — базова безпека WP-плагіна, що живе в недовіреному оточенні.
- **Alternatives considered**: (1) Показати бренд/лого CRM у меню плагіна для впізнаваності — відкинуто: порушує нейтральну ідентичність (01-overview) і Принцип II (не розкривати зв'язок із централізованою панеллю). (2) Fallback-константа з адресою бекенду в коді на випадок порожньої конфігурації — відкинуто прямо: заборонено Принципом II/FR-024. (3) raw cURL/file_get_contents для запиту — відкинуто: wp_remote_post — стандарт WP, поважає проксі/фільтри WP і коректніше на чужих хостингах. (4) Детальні повідомлення про помилку з адресою/кодом бекенду в тості 'Проверить соединение' — відкинуто: FR-032 забороняє розкривати топологію; показуємо нейтральне 'не вдалося підключитися'.

**Open notes (deferred to plan / implementation):** Точні параметри вікна толерантності timestamp (пропозиція ±300 с, узгодити з бекендом/проксі під FR-027) та політику anti-replay nonce (зберігання/TTL — виключно на бекенді; плагін лише генерує унікальний nonce на запит) закріпити на етапі плану/контракту. Ім'я/форму заголовків (X-SD-*) і схему тіла зафіксувати в спільному HTTP-контракті (єдиний дозволений канал взаємодії, Принцип I) — плагін і бекенд мають узгодити канонічний рядок бай-байтово. Опційний override секрету/endpoint через wp-config константи vs Options API — рішення розгортання; на чужих шаред-хостингах env-сховище не гарантоване, тому дефолт — Options API з autoload='no' і забороною повторного показу секрету. Надійність WP-Cron на тихих сайтах (ризик хибного офлайну проти SC-003) вимагає документованої інструкції з системного cron у вкладці 'Подключение' — врахувати як задачу UX/докам, а не лише код.

## Proxy / Ingress + Security

### Wire format: identity + signature headers

- **Decision**: Plugin sends every request with a fixed set of HTTP headers, none of which carry the secret: X-DB-Site-Id (public, non-secret site identifier — opaque, high-entropy, used by backend to look up the site + its secret), X-DB-Timestamp (Unix epoch seconds, integer), X-DB-Nonce (128-bit CSPRNG value, base64url, unique per request), X-DB-Signature (lowercase hex HMAC-SHA256), and X-DB-Sig-Version (scheme/version tag, e.g. "v1"). Content-Type application/json. The site-id and the HMAC secret are two distinct values (per clarify 2026-07-21): rotating the secret never changes the site-id.
- **Rationale**: Розділення публічного site-ідентифікатора і секретного HMAC-ключа прямо реалізує уточнення 2026-07-21 та FR-002/FR-003: id потрібен проксі й бекенду для пошуку сайту (несекретний), секрет ніколи не йде у відкриту (FR-004). Nonce з CSPRNG (128 біт) робить колізії неможливими на практиці й дає антиреплей у межах вікна. Явна версія схеми (X-DB-Sig-Version) дає крипто-аджилність — зміну алгоритму/канонізації без ламання старих плагінів. Заголовки, а не тіло, дозволяють проксі робити дешеву несекретну перевірку присутності/формату без парсингу тіла (FR-027).
- **Alternatives considered**: Єдиний bearer-токен (відкинуто: не дає цілісності тіла й антиреплею, а компрометація токена = повний доступ). Підпис у тілі JSON (відкинуто: змушує проксі парсити тіло для edge-перевірки, дорожче й крихкіше). JWT з підписом (відкинуто як надмірний: несе зайві поля, спокушає класти секрет/ключ на межу; простий HMAC над канонічним рядком легший для vanilla-PHP плагіна — Принцип V).

### Canonical string + HMAC-SHA256 computation

- **Decision**: Signature = HMAC-SHA256(secret, canonical_string), hex-encoded. canonical_string is a version-prefixed, newline-joined, length-unambiguous concatenation: "v1" \n HTTP-METHOD(upper) \n request-path (normalized, incl. sorted+encoded query if any) \n LOWERHEX(SHA-256(raw_request_body)) \n timestamp \n nonce. The raw body is bound via its SHA-256 hash rather than inlined verbatim, so the canonical string stays bounded and delimiter-injection-safe while still covering byte-exact body content (spec FR-011 'raw-body' intent is met — the body is authenticated, just via its digest). Backend and plugin MUST agree byte-for-byte on normalization; backend compares signatures in constant time.
- **Rationale**: Прив'язка методу+шляху+тіла+часу+nonce під один підпис виконує FR-011 (захист від підробки й від replay/зміни). Хешування тіла (а не вклеювання сирих байтів) прибирає ризик ін'єкції роздільників і робить канонічний рядок фіксованої довжини — важливо для WP-плагіна на vanilla-PHP, де тіло може бути великим. Версійний префікс і явна нормалізація шляху усувають розбіжності канонізації між PHP-плагіном і Laravel-бекендом. Порівняння підпису в constant-time (hash_equals) блокує тайминг-атаки. HMAC-SHA256 — галузевий стандарт (AWS SigV4, Stripe/GitHub webhooks) і є в ext-hash PHP без зовнішніх залежностей (Принцип V, III).
- **Alternatives considered**: Вклеювання сирого тіла в канонічний рядок (як буквально в ТЗ) — робочий варіант, але відкинуто на користь дайджесту тіла через ризик ін'єкції роздільників і необмежену довжину; семантично еквівалентно (тіло все одно автентифікується). HMAC-SHA512 (відкинуто: зайва довжина без реальної переваги). Асиметричний підпис Ed25519 (відкинуто: приватний ключ на 500+ чужих серверах = гірша модель; симетричний per-site секрет + миттєве відкликання простіший і відповідає FR-005).

### Timestamp tolerance window + nonce replay store

- **Decision**: Accept a request only if |now - X-DB-Timestamp| <= 300 s (±5 minutes). Reject outside the window as replay/clock-skew (FR-011, edge case 'мітка з майбутнього/застаріла'). Anti-replay within the window: a nonce store keyed by (site-id, nonce); first-seen is accepted and stored, any repeat within its TTL is rejected. Nonce TTL = 2× window = 600 s (must cover the full ±window acceptance range, not 1× — a captured request stays 'in-window' for up to 2W). Store MUST be atomic set-if-absent (e.g. Redis SET NX EX 600). The authoritative nonce check lives on the backend; the proxy only checks the timestamp window (it has no shared, consistent nonce state and must not be trusted for replay defense — FR-028).
- **Rationale**: ±300 с — консенсусне значення (AWS SigV4, багато webhook-схем) і достатнє для розсинхрону годинника на шаред-хостингу WordPress, де NTP не гарантований; ширше вікно збільшує площу для replay, вужче — ламає легальні звіти при дрейфі годинника. TTL nonce саме 2× вікна — тонкий, але критичний момент: запит із timestamp = now-300 приймається зараз і ще ~300 с матиме валідний час, тож nonce треба тримати 600 с, інакше з'явиться діра для повтору. Атомарний SET NX усуває гонки при паралельних звітах (edge case 'сотні сайтів одночасно'). Redis природно лягає поряд із Laravel Queue (Принцип IV) і дає TTL 'із коробки'.
- **Alternatives considered**: Монотонний лічильник/послідовність замість nonce (відкинуто: крихко при кількох інстансах плагіна, втраті стану, переустановці; nonce+timestamp stateless на боці плагіна). Вікно ±15 хв (відкинуто: занадто широке для 1/хв heartbeat). TTL = 1× вікна (відкинуто як помилкове — лишає 300-с діру для повтору). Nonce-стор у SQL-таблиці (припустимо, але дорожче й потребує ручного GC; Redis із TTL кращий за навантаження 500+ сайтів).

### Defense-in-depth split: proxy edge vs backend authoritative

- **Decision**: Proxy performs ONLY cheap, NON-SECRET, stateless edge checks and NEVER holds any per-site HMAC secret: (1) presence + syntactic format of X-DB-Site-Id, X-DB-Timestamp, X-DB-Nonce, X-DB-Signature, X-DB-Sig-Version; (2) timestamp within ±300 s; (3) request structural sanity (method, content-type, body size cap, header size); (4) per-site-id rate-limiting/quota. It forwards survivors to the backend and drops obvious garbage early. The CRM backend is the SOLE source of truth: it looks up the site by id, loads the per-site secret, recomputes HMAC and compares constant-time, independently re-checks the timestamp window AND the nonce store, and re-checks token state (active/revoked) and per-site data isolation. Backend treats the request as unauthenticated until ITS OWN crypto check passes and never trusts any proxy-added header as proof of authentication.
- **Rationale**: Прямо реалізує FR-027/028/029/031 і секцію 'Вимоги безпеки' конституції. Ключовий інваріант: секрет і його перевірка існують лише на бекенді, тому компрометація проксі не дає підробити валідний підпис — радіус ураження проксі обмежений (FR-031, Принцип II 'blast radius'). Проксі робить рівно ту роботу, що не потребує секрету й дешева на межі (відсіює сміття, throttling), а бекенд робить авторитетну крипто-перевірку та повторно все переперевіряє (не довіряє проксі наосліп — FR-028). Це і є defense-in-depth: два незалежні контури, жоден не є єдиною точкою відмови автентифікації.
- **Alternatives considered**: Перевірка HMAC на проксі/edge (категорично відкинуто: вимагає роздати per-site секрети на межу — компрометація проксі = злив секретів усієї мережі, пряме порушення FR-027/029 і 'Вимог безпеки'). Проксі додає 'X-Authenticated: true' і бекенд йому вірить (відкинуто: порушує FR-028, робить проксі єдиною точкою довіри). Лише бекенд без edge-перевірок (відкинуто: втрачається дешеве раннє відсіювання й throttling до бекенду, FR-026/027).

### Rate-limiting key and budget

- **Decision**: Rate-limit key = X-DB-Site-Id (the public site identifier), NOT client IP. Algorithm: token bucket per site-id. Recommended proxy budget for a ~1/min heartbeat: sustained ~10 requests/min with burst capacity 20 (refill ≈1 token/6 s). Over-limit → HTTP 429 with Retry-After, dropped BEFORE reaching the backend (FR-026). The backend keeps an independent, slightly tighter second-tier limit (e.g. sustained 6/min per site) as the second contour. Requests with a malformed/absent site-id (can't be keyed) fall into a small shared 'unkeyed' bucket and are aggressively throttled.
- **Rationale**: Ключем має бути site-id, бо плагіни стоять на шаред-хостингу — багато сайтів за одним вихідним IP, тож ліміт по IP або карав би сусідів, або був би марно широким (спец. вимога задачі). Бюджет ~10/хв при heartbeat 1/хв дає ~10× запас на ретраї, ресинк після офлайну і мережеві повтори, але жорстко ріже флуд/DoS з одного скомпрометованого плагіна (Принцип II — обмеження blast radius). Token bucket згладжує сплески краще за фіксоване вікно. Незалежний другий контур на бекенді виконує FR-026 (проксі не єдиний рубіж). 429 з Retry-After коректно керує поведінкою плагіна без розкриття топології.
- **Alternatives considered**: Ключ по IP (відкинуто: шаред-хостинг, FR каже прямо site-id). Fixed-window лічильник (відкинуто: дозволяє подвійний сплеск на межі вікон; token bucket рівніший). Sliding-window log (точніший, але дорожчий по пам'яті на 500+ ключів — token bucket достатній). Ліміт лише на бекенді (відкинуто: не відсіює трафік до бекенду, порушує ідею FR-026).

### TLS on both legs + backend network isolation

- **Decision**: Leg 1 (plugin↔proxy): TLS terminated at the proxy, TLS 1.2 minimum, TLS 1.3 preferred; HSTS; valid publicly-trusted cert on the single public endpoint. Leg 2 (proxy↔backend): NEVER cleartext — re-encrypt with TLS to the origin AND use mutual TLS (mTLS) so the backend only accepts connections presenting the proxy's client cert. The CRM backend has NO public IP / is not routable from the internet: private network/VPC, firewall/security-group allowlist permitting inbound only from the proxy, its host/port never disclosed. mTLS on leg 2 authenticates the transport channel but does NOT substitute for per-request HMAC re-verification (defense-in-depth — backend still verifies every request cryptographically).
- **Rationale**: Виконує FR-020/FR-025 і 'Вимоги безпеки' конституції: TLS на ВСЬОМУ шляху, термінація на проксі не створює відкритої внутрішньої ділянки. mTLS на внутрішній нозі — сильніший за простий re-encrypt: бекенд приймає з'єднання лише від легітимного проксі, звужуючи поверхню атаки, навіть якщо приватна мережа скомпрометована. Приватна мережа + allowlist реалізує FR-023 (бекенд не досяжний публічно, топологія прихована). Важливо: mTLS автентифікує канал, але не запит — тому FR-028 лишається чинним, бекенд усе одно перевіряє HMAC (mTLS ≠ підстава довіряти вмісту запиту).
- **Alternatives considered**: Re-encrypt TLS без mTLS + мережева ізоляція (прийнятний мінімум, якщо mTLS операційно важкий; слабший — не автентифікує сам проксі на рівні каналу). Внутрішня нога у відкриту в 'довіреній' мережі (категорично відкинуто: порушує 'жодної незашифрованої ділянки'). VPN/service-mesh (WireGuard, Istio) замість прямого mTLS (еквівалентно за гарантіями; вибір — питання інфраструктури на етапі плану, не специфікації).

### Product-agnostic proxy selection (no vendor pin)

- **Decision**: Do NOT pin a product (spec/constitution require product-agnosticism). Three viable classes, each acceptable if it meets the criteria: (A) self-hosted reverse proxy — nginx, Caddy, Traefik, HAProxy; (B) API gateway — Kong, KrakenD, APISIX, Envoy, or managed AWS API Gateway / Azure APIM; (C) CDN with edge logic — Cloudflare (Workers), Fastly (Compute), CloudFront + Lambda@Edge. Mandatory selection criteria: (1) can rate-limit keyed on a request HEADER value (site-id), not just IP; (2) can do cheap header/timestamp/structure validation at the edge WITHOUT holding any secret; (3) TLS termination + re-encrypt or mTLS to a private origin; (4) MUST NOT require placing the per-site HMAC secret at the edge; (5) audit logging of edge events (rejections, 429, WAF hits) correlatable with backend audit log (FR-033); (6) optional WAF/filtering (FR-030); (7) hides origin address/topology (FR-023/032). Secondary: self-hostability / data-locality, cost, and ops familiarity — a Laravel/self-hosted shop leans toward nginx/Traefik/Caddy; a cloud-native deploy may prefer Cloudflare or a managed gateway.
- **Rationale**: Специфікація (OQ-4) і конституція прямо вимагають продукт-агностичності — вибір вендора це рішення етапу плану, не Phase-0. Головна відсіювальна вимога — секрет НЕ на межі (критерій 4): це виключає будь-яку конфігурацію, де edge-логіка мала б перевіряти HMAC, і зберігає FR-027/031. Rate-limit по заголовку (критерій 1) обов'язковий через шаред-хостинг (див. рішення про ключ лімітів). Корельований аудит (критерій 5) виконує FR-033. Перелік трьох класів із критеріями дає команді рамку вибору без нав'язування продукту й без порушення Принципу III (для self-hosted варіантів — жодних зовнішніх runtime-звернень з чужих сайтів).
- **Alternatives considered**: Зафіксувати конкретний продукт зараз (відкинуто: пряме порушення вимоги агностичності специфікації/конституції; передчасно). Обійтися без проксі, лише бекенд за firewall (відкинуто: OQ-4 вирішено — проксі обов'язковий, FR-023..033). CDN-only без re-encrypt до приватного origin (відкинуто, якщо не забезпечує критерій 3). Managed-gateway у хмарі варто зважити проти вимоги локальності залежностей і того, що edge не тримає секретів.

### Secret storage, comparison, and rotation semantics

- **Decision**: Per-site HMAC secret: 256-bit CSPRNG value, shown to the operator exactly once at registration/rotation, then stored on the backend ENCRYPTED-AT-REST in a recoverable form (app-level/envelope encryption via KMS or Laravel encrypter, key in env/secret store — NOT in repo), because HMAC verification requires recomputing with the actual secret (it canNOT be one-way hashed like a password). Signature comparison uses constant-time hash_equals. State machine: active / revoked. Revocation takes effect near-immediately (backend rejects any request whose site is revoked/rotated); rotation issues a new secret WITHOUT changing the public site-id. Optionally support a short dual-secret overlap window (accept old+new) so a plugin re-config doesn't drop the site to offline mid-rotation.
- **Rationale**: Виконує FR-002/003/004/005 і 'Вимоги безпеки'. Критичний нюанс, який треба зафіксувати: конституція каже 'хешованими на боці CRM', але для HMAC бекенд ОБОВ'ЯЗКОВО потребує сам секрет, щоб перерахувати підпис — тому це шифрування з можливістю розшифрування (envelope/KMS), а не bcrypt/односторонній хеш. 256 біт — стандартна ентропія для HMAC-ключа. Constant-time порівняння обов'язкове проти тайминг-атак. Ротація без зміни id прямо реалізує уточнення 2026-07-21. Вікно перекриття двох секретів згладжує ротацію, щоб сайт не впав в офлайн (SC-002/003) поки вебмайстер оновлює конфіг плагіна.
- **Alternatives considered**: Односторонній хеш секрету (неможливо — HMAC потребує оригінал; годиться лише якби замість підпису був простий bearer-токен). Зберігати секрет у відкриту в БД (відкинуто: витік БД = злив усієї мережі). Без вікна перекриття при ротації (простіше, але дає короткий офлайн сайту під час зміни ключа — прийнятно для MVP, але гірший UX).

**Open notes (deferred to plan / implementation):** Питання/уточнення для етапу плану (plan.md) і конфігурації:

1. РОЗБІЖНІСТЬ ФОРМУЛЮВАННЯ (треба узгодити в плані): конституція, секція 'Вимоги безпеки', каже зберігати токени/секрети 'хешованими на боці CRM'. Для HMAC це неможливо — бекенд потребує оригінал секрету, щоб перерахувати підпис. Правильна вимога: секрет шифрується з можливістю розшифрування (envelope/KMS/Laravel encrypter, ключ у env/сховищі секретів), а НЕ односторонньо хешується. Рекомендую уточнити формулювання конституції/плану, щоб не було хибної реалізації через bcrypt.

2. Конкретні числові дефолти закладені як рекомендації й мають стати конфігом (A-2 вже позначає їх налаштовуваними): вікно толерантності часу ±300 с; TTL nonce 600 с (=2× вікна — навмисно, не 1×); rate-limit проксі ~10/хв burst 20 per site-id, бекенд ~6/хв. Точні значення закріпити в plan/config.

3. Технологія nonce-стору: рекомендований Redis (SET NX EX), що природно співіснує з Laravel Queue (Принцип IV). Якщо кілька інстансів бекенду — стор має бути спільним/атомарним. Підтвердити на етапі плану наявність Redis у стеку.

4. Внутрішня нога проксі↔бекенд: рекомендовано mTLS; якщо операційно важко — мінімум re-encrypt TLS + приватна мережа + allowlist. Вибір інфраструктури (VPN/service-mesh vs прямий mTLS) — рішення плану.

5. Вибір продукту проксі свідомо НЕ зроблено (вимога агностичності). Наведено 3 класи + 7 критеріїв відбору; фінальний вибір — етап плану. Головний відсіювач: продукт НЕ повинен вимагати розміщення per-site секрету на межі.

6. Канонічний рядок: обрано варіант із дайджестом тіла (SHA-256) замість вклеювання сирих байтів, попри буквальне 'raw-body' у ТЗ — семантично тіло все одно автентифіковане, але усунуто ін'єкцію роздільників і необмежену довжину. Якщо команда наполягає на буквальному raw-body, зафіксувати length-prefixed join, щоб уникнути неоднозначності канонізації між vanilla-PHP плагіном і Laravel.

7. Ротація секрету: опційне вікно перекриття двох секретів (old+new) рекомендоване, щоб уникнути короткого офлайну сайту під час переконфігурації плагіна (впливає на SC-002/003). Для MVP допустимо без нього — зважити UX vs складність у плані.

8. Кореляція аудиту (FR-033): потрібен спільний correlation-id (напр. з X-DB-Nonce або окремий request-id), що логується і на проксі, і на бекенді, щоб edge-події співвідносилися з аудит-логом CRM.

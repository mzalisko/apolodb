## 2. Передумови

### 2.1. CRM-бекенд (стек)

- **PHP 8.3–8.5**, **Laravel 13.x**
- **PostgreSQL 16+** (окрема БД CRM, не БД WordPress)
- **Redis** — черга, кеш, nonce-стор антиреплею, per-site rate-limit
- **Laravel Horizon** — моніторинг воркерів черги (перевірка асинхронності, SC-007)
- **Node.js + npm + Vite** — збірка Blade-фронтенду (Tailwind/Alpine self-hosted, без CDN)
- Встановлені залежності: `composer install`, `npm ci`
- `.env` сконфігуровано (див. розділ 2.4), виконано `php artisan migrate`

### 2.2. Проксі / ingress-шлюз (обов'язковий компонент)

Єдина публічна точка приймання перед бекендом. Для локальної валідації підійде будь-який із класів
(nginx / Caddy / Traefik / API-gateway / CDN з edge-логікою) за умови, що він:

- термінує TLS на межі та **повторно** шифрує канал до бекенда (re-encrypt або mTLS);
- застосовує per-**site-id** rate-limiting (ключ = заголовок site-id, **не** IP);
- виконує лише **дешеві несекретні** edge-перевірки (наявність/формат заголовків, вікно timestamp);
- **не** тримає жодного per-site HMAC-секрету і **не** перевіряє HMAC сам;
- журналює події межі з correlation-id, співвідносним із аудит-логом бекенда.

Бекенд **не** має бути публічно досяжним — лише через проксі (FR-023). Для локального прогону
достатньо, щоб плагін/`curl` звертались на публічний ендпоінт проксі, а проксі проксіював на бекенд.

### 2.3. Тестовий WordPress-сайт із плагіном

- WordPress із встановленим плагіном цієї фічі (vanilla-PHP, WP Options API).
- Доступ до вкладки налаштувань «Подключение» (нейтральна ідентичність, пункт меню «Данные сайта»).
- Можливість запустити системний cron для `wp-cron.php` (для «тихого» сайту — див. S2).

### 2.4. Ключові дефолти конфігурації (env / config)

Значення налаштовувані; нижче — **базові дефолти** цієї фічі. Точні межі фіксуються конфігом.

| Параметр | Дефолт | Призначення |
|----------|--------|-------------|
| Інтервал heartbeat плагіна | ~60 с | частота звітів (A-2) |
| Вікно офлайну | ~5 хв (3 пропущені звіти) | коли сайт стає «офлайн» (SC-003) |
| Вікно толерантності timestamp | ±300 с | антидрейф годинника + межа replay (FR-011) |
| TTL nonce (антиреплей) | 600 с (**2×** вікна) | покриває весь діапазон прийняття запиту |
| Rate-limit на проксі | ~10/хв на site-id, burst 20 | перший контур (FR-026) |
| Rate-limit на бекенді | ~6/хв на site-id | другий, незалежний контур (FR-026) |
| Алгоритм підпису | HMAC-SHA256 | автентифікація запиту (A-4) |

## 3. Запуск системи (високорівнево)

Для наскрізної валідації мають одночасно працювати **чотири** процеси бекенда + проксі.

```bash
# 1) Бекенд (за проксі; для локалу — на приватному інтерфейсі)
php artisan serve            # або php-fpm/nginx у бойовій конфігурації

# 2) Воркер черги — обробляє ProcessHeartbeat (ідемпотентний upsert статусу)
php artisan queue:work redis --queue=default
#   або, для видимості глибини черги/throughput:
php artisan horizon

# 3) Планувальник — щохвилини запускає детектор офлайну (sites:detect-offline)
#    У проді: системний cron/systemd-timer, що викликає schedule:run щохвилини.
php artisan schedule:work    # локальний еквівалент постійного schedule:run

# 4) Фронтенд-асети (одноразова збірка або watch)
npm run build                # self-hosted CSS/JS/шрифти, без CDN
```

Проксі піднімається окремо своїм способом і має вказувати upstream на приватний адрес бекенда.

**Перевірка готовності**: `php artisan horizon:status` показує `running`; `php artisan schedule:list`
містить `sites:detect-offline` з періодом `everyMinute` та `withoutOverlapping`.

## 4. Наскрізні сценарії валідації

> У прикладах `PROXY_URL` — публічний ендпоінт проксі (напр. `https://ingress.example`),
> `SITE_ID` та `SECRET` — видані на кроці S1. Точний шлях приймання, назви заголовків та байт-точний
> канонічний рядок беруться з [`contracts/`](./contracts/) — нижче використано їх як ілюстрацію.

### S1 — Реєстрація сайту → site-id + одноразовий секрет

**Мета**: FR-001..FR-004, FR-006, SC-001 — оператор реєструє сайт і отримує облікові дані.

**Кроки**

1. Увійти в CRM як адміністратор, відкрити список сайтів → «Додати сайт».
2. Вказати назву й домен (напр. `example.com`) → зберегти.
3. Система показує **один раз**: публічний `site-id` (несекретний) і секретний `HMAC-секрет`.
   Скопіювати обидва.

**Очікуваний результат**

- Сайт з'явився у списку зі статусом **«очікує» (синій)**.
- Секрет показано **рівно один раз**; після закриття вікна відображається лише маска (`••••`),
  повторний перегляд секрету неможливий — лише **перевипуск** (FR-004).
- `site-id` доступний для повторного перегляду (несекретний).
- Спроба додати **той самий домен** повторно → відмова без створення дубля (FR-006).
- В аудит-лозі з'явився запис події «Токен»: хто, коли, було → стало (FR-021).

**Швидка перевірка (за потреби, read-only через artisan tinker)**

```bash
php artisan tinker
>>> \App\Models\Site::where('domain','example.com')->first()->status;   // "pending"
```

### S2 — Конфігурація плагіна → сайт «онлайн» протягом ~1 хв

**Мета**: FR-007, FR-009, FR-010, FR-012, SC-002.

**Кроки**

1. У плагіні відкрити вкладку «Подключение».
2. Ввести: `endpoint URL` = `PROXY_URL` (публічний ендпоінт проксі), `site-id`, `HMAC-секрет`.
   Зберегти. Секрет записується, але назад у поле **не** рендериться.
3. Переконатися, що WP-Cron активний. Для «тихого» сайту без трафіку — увімкнути системний cron:
   вимкнути псевдо-крон (`define('DISABLE_WP_CRON', true)`) і викликати `wp-cron.php` щохвилини
   (crontab або `wp cron event run --due-now`), щоб heartbeat надсилався регулярно.
4. За бажанням натиснути «Проверить соединение» — плагін надсилає підписаний запит одразу.

**Очікуваний результат**

- Протягом **~1 хвилини** (один інтервал) сайт у списку CRM переходить у **«онлайн» (зелений)**.
- Час «останнього оновлення» сайту оновлюється на момент прийнятого звіту (FR-012).
- Звіт оброблено **через чергу** — у Horizon видно виконані джоби `ProcessHeartbeat`, приймання
  повернуло 202/204 одразу, обробка асинхронна (FR-010).

**Перевірка**

```bash
php artisan tinker
>>> \App\Models\Site::where('domain','example.com')->first()->status;         // "online"
>>> \App\Models\Site::where('domain','example.com')->first()->last_seen_at;   // ~щойно
```

### S3 — Зупинка плагіна → сайт «офлайн» протягом ~5 хв

**Мета**: FR-013, FR-014, SC-003 — офлайн визначається **тишею** (push-based), CRM не пінгує сайт.

**Кроки**

1. Зупинити надсилання heartbeat: вимкнути плагін / зупинити системний cron сайту.
2. Не втручатися; лишити планувальник CRM працювати (`sites:detect-offline` щохвилини).

**Очікуваний результат**

- Протягом **~5 хвилин** (3 пропущені звіти) від часу останнього прийнятого звіту сайт переходить
  у **«офлайн» (червоний)**.
- Перехід `online → offline` зафіксовано set-based командою `sites:detect-offline` (один UPDATE),
  а не пінгом сайту (FR-014).
- У changelog з'явився запис про транзицію статусу (FR-021).

**Перевірка**

```bash
# Ручний прогін детектора (не чекати планувальник):
php artisan sites:detect-offline
php artisan tinker
>>> \App\Models\Site::where('domain','example.com')->first()->status;   // "offline"
```

> Симетрія: щойно плагін відновить звіти, найближчий heartbeat-upsert поверне сайт у «онлайн»
> без окремої команди.

### S4 — Неавтентифікований / поганий підпис / replay → відхилено

**Мета**: FR-007, FR-011, FR-027, FR-028, SC-004, SC-005. Доводить, що бекенд авторитетно
перевіряє кожен запит і не довіряє факту проходження через проксі.

Байт-точні заголовки/канонік — у [`contracts/`](./contracts/). Нижче — три негативні кейси.

**4a. Без облікових даних (неавтентифікований)**

```bash
curl -i -X POST "$PROXY_URL/heartbeat" \
  -H "Content-Type: application/json" \
  -d '{"site_id":"'"$SITE_ID"'","status":"online","timestamp":'"$(date +%s)"',"nonce":"deadbeef"}'
```

*Очікування*: відхилення (`401`/`400`) — відсутній підпис. Статус сайту **не** змінюється.

**4b. Поганий підпис**

Сформувати валідні заголовки, але з підписом від **невірного** секрету (напр. `WRONG_SECRET`).

*Очікування*: бекенд перераховує HMAC із **правильним** per-site секретом, constant-time порівняння
не збігається → відхилення (`401`). Запит **не** впливає на статус (FR-028). Проксі при цьому не
робить HMAC-перевірку — рішення ухвалює виключно бекенд.

**4c. Replay та застарілий/майбутній timestamp**

- Повторно надіслати **раніше валідний** запит (той самий nonce, у межах вікна) → відхилення:
  nonce вже у стор і (site-id, nonce) повторюється (FR-011, SC-005).
- Надіслати запит із `timestamp` поза вікном ±300 с (застарілий або з майбутнього) → відхилення
  ще на межі (проксі, FR-027) та/або на бекенді (FR-028).

*Очікування (усі 4a–4c)*: **100%** таких запитів відхилено; статус і час останнього оновлення
сайту незмінні. Перевірити журнал приймання/аудит межі — записи про відхилення з correlation-id
(FR-033).

### S5 — Відкликання секрету → підписані запити відхиляються

**Мета**: FR-005, FR-029, SC-006 — скомпрометований секрет знешкоджується майже одразу per-site.

**Кроки**

1. У CRM відкрити сайт → «Відкликати токен».
2. Не змінюючи конфіг плагіна (він досі підписує **старим** секретом), дочекатися наступного
   heartbeat або натиснути «Проверить соединение».

**Очікуваний результат**

- Будь-який запит зі **старим** секретом відхиляється майже одразу (у межах ~1 хв поширення, SC-006),
  бо стан токена на бекенді = `revoked`; `site-id` при цьому не змінюється (FR-005).
- Сайт перестає отримувати підтвердження і з часом (вікно офлайну) позначається **«офлайн»**.
- Перевипуск нового секрету (показ один раз) + введення його в плагін → звіти знову приймаються,
  сайт повертається в «онлайн». `site-id` лишається тим самим (ротація секрету не змінює ідентифікатор).

**Перевірка**

```bash
php artisan tinker
>>> $s = \App\Models\Site::where('domain','example.com')->first();
>>> $s->credential->state;    // "revoked" до перевипуску
```

### S6 — Список сайтів зі статусом і часом останнього оновлення

**Мета**: FR-015, FR-016, FR-017, FR-018 — головний екран оператора.

**Кроки**

1. Мати кілька сайтів у різних станах (онлайн / очікує / офлайн / неактивний).
2. Відкрити головний екран — список сайтів.

**Очікуваний результат**

- Кожен рядок: сайт/домен, поточний статус за **єдиною легендою кольорів**
  (онлайн→зелений, очікує→синій, офлайн→червоний, неактивний→сірий; FR-016) і час останнього
  оновлення (A-7). Бурштиновий «резерв» до сайтів **не** застосовується.
- Список рендериться сервером (Blade), self-hosted асети, **жодних** зовнішніх CDN-звернень
  (перевірити вкладку Network — усі ресурси з власного домену).
- Фільтр за статусом (напр. «офлайн») показує лише відповідні сайти з лічильником **«N із M»** (FR-018).
- Оновлення **не** зобов'язане бути real-time; після нового звіту статус оновлюється при
  перезавантаженні/легким Alpine-фільтром (FR-017).

## 7. Типові проблеми

- **Сайт не стає «онлайн»**: WP-Cron не спрацьовує на «тихому» сайті — увімкнути системний cron для
  `wp-cron.php` (S2, крок 3). Перевірити, що `endpoint URL` вказує на **проксі**, а не на бекенд.
- **Усі heartbeat відхиляються (401)**: розсинхрон годинника поза ±300 с — синхронізувати NTP; або
  розбіжність канонічного рядка плагін↔бекенд — звірити з [`contracts/`](./contracts/) байт-у-байт.
- **Сайт не переходить в «офлайн»**: не працює планувальник — переконатися, що системний cron
  викликає `schedule:run` щохвилини і команда `sites:detect-offline` присутня у `schedule:list`.
- **Джоби не обробляються**: не запущено `queue:work`/`horizon`, або Redis недоступний.
```

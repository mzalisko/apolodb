# Контракти інтерфейсів — Feature 001: Реєстрація сайтів та моніторинг статусу підключення

**Feature**: `001-site-connectivity-failover`
**Статус**: Contract (Phase-1)
**Дата**: 2026-07-21
**Обсяг**: покриває FR-001..FR-033. Документ описує **контракти** (форми запитів/відповідей, поля, семантику, коди статусів, крипто-правила), а не реалізацію. Вибір конкретних продуктів (проксі/ingress, СУБД, черга, крипто-бібліотека) навмисно **не фіксується** — контракт продукт-агностичний.

---

## 0. Загальні положення

### 0.1 Учасники та канали

| Учасник | Роль | Що знає |
|---|---|---|
| **Plugin** (WordPress, vanilla PHP) | Клієнт-відправник. Періодично підписує й надсилає heartbeat. | Лише публічний `site-id`, `signing-secret`, `endpoint-url` (один публічний ендпоінт проксі). Не знає топології бекенду (FR-008, FR-024). |
| **Proxy / Ingress Gateway** | Єдина публічна точка приймання. Дешева **несекретна** edge-перевірка, rate-limiting, TLS-термінація, приховування топології. | **НЕ тримає** per-site HMAC-секретів (FR-027, FR-029, FR-031). |
| **CRM Backend** («Bridge») | Джерело істини. Авторитетна крипто-верифікація, стан токена, ізоляція даних. | Per-site секрети (encrypted-at-rest), стан токенів, дані сайтів. Не досяжний із публічної мережі (FR-023). |

Єдиний дозволений канал взаємодії plugin↔CRM — цей HTTP-контракт (Принцип I). Уся комунікація — виключно поверх TLS на **обох** ділянках (plugin↔proxy і proxy↔backend), жодної відкритої ділянки (FR-020, FR-025).

### 0.2 Конвенції

- **Кодування тіла**: `application/json; charset=utf-8`. Числа часу — Unix epoch **секунди** (UTC, ціле).
- **Префікс заголовків**: `X-DB-*` (DataBridge). Це канонічна схема цього контракту; плагін і бекенд ПОВИННІ узгодити її **байт-у-байт**.
- **Версія схеми підпису**: рядок-тег `v1` (заголовок `X-DB-Sig-Version`) для крипто-аджилності — зміна алгоритму/канонізації не ламає старих клієнтів.
- **Нейтральність помилок**: жодна відповідь проксі чи бекенду НЕ розкриває внутрішню адресу/топологію, існування чи ідентифікатори інших сайтів (FR-019, FR-032). Тіла помилок — узагальнені (див. 2.4).

### 0.3 Конфігуровані параметри (базові дефолти)

Значення налаштовувані (config/env), тут наведено дефолти контракту:

| Параметр | Дефолт | Джерело |
|---|---|---|
| Інтервал звітування плагіна | ~60 с | A-2, FR-009 |
| Вікно офлайну | 3 пропущені звіти (~300 с) | A-2, FR-014, SC-003 |
| Timestamp tolerance window `W` | ±300 с | A-4, FR-011 |
| Nonce TTL | 600 с (= 2×W) | anti-replay |
| Rate-limit на проксі (per site-id) | sustained ~10/хв, burst 20 | FR-026 |
| Rate-limit на бекенді (per site-id, 2-й контур) | sustained ~6/хв | FR-026 |
| Максимальний розмір тіла heartbeat | 8 KiB | edge sanity, FR-027 |

## 1. Контракт підписаного запиту (plugin → proxy → backend)

Застосовується до **кожного** запиту від плагіна до точки приймання (наразі — heartbeat, розділ 2). CRM-internal endpoints (розділ 3) цим контрактом **не** покриваються — вони автентифікуються сесією/токеном оператора.

### 1.1 Обов'язкові заголовки

| Заголовок | Тип / формат | Секретний? | Призначення |
|---|---|---|---|
| `Content-Type` | `application/json; charset=utf-8` | ні | тип тіла |
| `X-DB-Site-Id` | opaque, high-entropy рядок | **ні** (публічний) | пошук сайту та його секрету на бекенді (FR-002) |
| `X-DB-Timestamp` | Unix epoch секунди, ціле | ні | прив'язка до часу, anti-replay (FR-011) |
| `X-DB-Nonce` | 128-біт CSPRNG, base64url, унікальний на запит | ні | anti-replay у межах вікна (FR-011) |
| `X-DB-Signature` | `sha256=<lowercase-hex>` | ні (похідний) | HMAC-SHA256 підпис каноніка |
| `X-DB-Sig-Version` | тег схеми, напр. `v1` | ні | крипто-аджилність |

**Секрет (`signing-secret`) у запиті НЕ передається в жодному вигляді** — ані в заголовках, ані в тілі. `site-id` і `signing-secret` — два різні значення; ротація секрету не змінює `site-id` (A-4, FR-002).

### 1.2 Канонічний рядок (те, що підписується)

Канонічний рядок — версіонований, з фіксованим порядком полів, з'єднаних символом `\n` (LF, `0x0A`), **без** кінцевого переводу рядка. Сире тіло прив'язується через його SHA-256-дайджест (а не вклеюється дослівно) — це тримає рядок обмеженим і стійким до ін'єкції роздільників, але тіло лишається автентифікованим байт-у-байт (виконує намір FR-011 «raw-body»).

```
canonical =
  "v1"                              LF
  UPPER(HTTP-METHOD)                LF      // напр. "POST"
  canonical-request-path            LF      // нормалізований шлях; за наявності query — відсортований і url-encoded
  LOWERHEX( SHA-256( raw-body ) )   LF      // hex-дайджест точних байтів тіла
  site-id                           LF      // = X-DB-Site-Id
  timestamp                         LF      // = X-DB-Timestamp
  nonce                                     // = X-DB-Nonce (без кінцевого LF)
```

Правила:
- `raw-body` — **точні байти** тіла, що йдуть у HTTP-запит (плагін хешує саме те, що надсилає).
- Порядок полів фіксований і **не** сортується. Плагін і бекенд ПОВИННІ отримати ідентичний рядок байт-у-байт; будь-яка розбіжність нормалізації → невалідний підпис.
- `site-id`, `timestamp`, `nonce` у каноніку ДОРІВНЮЮТЬ значенням однойменних заголовків; підміна заголовка без переобчислення підпису → відхилення.

### 1.3 Обчислення підпису

```
signature = HMAC-SHA256( key = signing-secret, message = canonical )
X-DB-Signature: "sha256=" + LOWERHEX(signature)
```

- Алгоритм: **HMAC-SHA256**, вивід — lowercase hex.
- Плагін лише **підписує**; порівняння підпису робить бекенд у **constant-time** (див. 1.5).
- Плагін НЕ виконує жодної верифікації відповіді, крім читання коду статусу.

### 1.4 Вікно толерантності часу та правило anti-replay (nonce)

**Timestamp window** (FR-011, SC-005):
- Запит приймається лише якщо `|now − X-DB-Timestamp| ≤ W` (дефолт W = 300 с).
- Поза вікном (застаріла або з майбутнього мітка) → відхилення як replay/clock-skew.

**Nonce replay store** (авторитетно — **лише на бекенді**, FR-028):
- Ключ анти-реплей запису: пара `(site-id, nonce)`.
- Перше пред'явлення пари в межах TTL — приймається й зберігається; будь-який повтор у межах TTL — відхиляється.
- **Nonce TTL = 2×W = 600 с** (навмисно 2×, не 1×): запит із міткою `now − W` лишається «в вікні» ще до `W` секунд, тож nonce треба тримати весь діапазон прийнятності, інакше лишається діра для повтору.
- Запис ПОВИНЕН бути атомарним «set-if-absent» (семантика типу `SET NX EX 600`), щоб уникнути гонок за одночасних звітів.
- Проксі **не** має спільного консистентного nonce-стану й **не** є рубежем анти-реплею — він перевіряє лише timestamp-вікно (FR-028).

### 1.5 Дворівнева верифікація (defense-in-depth)

**Рівень 1 — Proxy (дешеві НЕСЕКРЕТНІ edge-перевірки, FR-026, FR-027, FR-030).** Проксі НІКОЛИ не тримає per-site секрет і виконує лише:
1. Присутність і синтаксичний формат `X-DB-Site-Id`, `X-DB-Timestamp`, `X-DB-Nonce`, `X-DB-Signature`, `X-DB-Sig-Version`.
2. Timestamp у межах ±W.
3. Структурна валідність: метод, `Content-Type`, ліміт розміру тіла, ліміт розміру заголовків.
4. Per-`site-id` rate-limiting / квоти (token-bucket; ключ — **значення заголовка `X-DB-Site-Id`, НЕ IP** — бо плагіни живуть на шаред-хостингу за спільними IP). Перевищення → `429` **до** пересилання на бекенд.
5. Опційна WAF-фільтрація.

Проксі відсіює очевидне сміття рано й пересилає «виживших» на бекенд. Проксі НЕ виконує HMAC-перевірку і НЕ додає жодного заголовка, що бекенд трактував би як доказ автентифікації.

**Рівень 2 — Backend (авторитетна перевірка, FR-028, FR-029, FR-031).** Бекенд вважає запит **неавтентифікованим**, доки не пройде його ВЛАСНА крипто-перевірка, і НЕ довіряє жодному доданому проксі заголовку. Бекенд:
1. Знаходить сайт за `X-DB-Site-Id`; якщо не знайдено або токен `revoked` → відхилення.
2. Завантажує per-site секрет, **переобчислює** канонічний рядок і HMAC, порівнює **constant-time**.
3. Незалежно перевіряє timestamp-вікно **та** nonce-стор (перше-пред'явлення).
4. Перевіряє стан токена (active/revoked) і per-site ізоляцію даних (FR-019, FR-029).
5. Лише за успіху всіх перевірок ставить heartbeat у чергу на обробку.

Компрометація проксі НЕ дає підробити валідний підпис і НЕ дає обійти рівень 2 — радіус ураження проксі обмежений (FR-031).

**Кореляція аудиту (FR-033).** Проксі журналює edge-події (відхилення, `429`, спрацювання WAF) у придатному до аудиту вигляді; для співвіднесення з аудит-логом бекенду використовується спільний correlation-id (значення `X-DB-Nonce` або окремий `X-DB-Request-Id`), що логується на обох боках.

## 2. Ingest endpoint — POST status heartbeat

Єдиний plugin-facing ендпоінт. Публічна адреса — ендпоінт **проксі** (плагін іншої адреси не знає, FR-024). Внутрішній шлях на бекенді топологічно прихований.

```
POST  {proxy-endpoint}/v1/heartbeat
```

Автентифікація — підписаний запит із розділу 1 (усі заголовки `X-DB-*` обов'язкові). Обробка — **асинхронна**: приймання підтверджується одразу, фактичний upsert статусу відбувається у черзі (FR-010, Принцип IV).

### 2.1 Схема запиту (тіло)

Компактний JSON рівно з полями цієї фічі:

```jsonc
{
  "site_id":   "<публічний site-ідентифікатор>",   // рядок; ДОРІВНЮЄ X-DB-Site-Id
  "status":    "online",                            // enum; для цієї фічі — лише "online"
  "timestamp": 1753080000,                          // Unix epoch секунди UTC; ДОРІВНЮЄ X-DB-Timestamp
  "nonce":     "<base64url 128-біт>"                // ДОРІВНЮЄ X-DB-Nonce
}
```

Правила схеми:
- `status` для цієї фічі приймає **лише `"online"`**. Плагін звітує тільки коли живий; `"offline"` плагін фізично надіслати не може — офлайн виводить бекенд за **відсутністю** звітів (push-based, FR-013, FR-014). Значення поза enum → відхилення.
- Поля `timestamp`/`nonce` дублюють однойменні заголовки для зручності бекенду; вони ПОВИННІ збігатися із заголовками (розбіжність → відхилення).
- **Підпис у тілі відсутній** — він лише в заголовку `X-DB-Signature` (інакше дайджест тіла залежав би від самого підпису).
- **Жодних** контактних даних (телефони, ланцюг резервних номерів, ціни, гео, версія плагіна тощо) — поза межами фічі (A-1).
- Невідомі поля ігноруються або відхиляються (політика — рішення плану); зайвий обсяг обмежений лімітом 8 KiB.

### 2.2 Успішна відповідь (accepted-async)

Приймання підтверджується **до** обробки (latest-wins upsert відбувається у черзі):

```
HTTP/1.1 202 Accepted        (або 204 No Content)
```

Тіло — порожнє або мінімальне підтвердження без будь-яких даних сайту:

```jsonc
{ "accepted": true }
```

Відповідь **не** повертає статус сайту, час, ідентифікатори чи будь-що про інші сайти (FR-019, FR-032). `202/204` означає «прийнято до обробки», а не «стан застосовано» (FR-010, FR-017).

### 2.3 Коди статусів (heartbeat)

| Код | Значення | Хто повертає | Умова |
|---|---|---|---|
| `202 Accepted` / `204 No Content` | Прийнято асинхронно | Backend | Усі перевірки рівня 2 пройдено; поставлено в чергу |
| `400 Bad Request` | Некоректна структура/JSON | Proxy або Backend | Невалідне тіло, відсутні/зламані обов'язкові поля/заголовки, `status` поза enum |
| `401 Unauthorized` | Не автентифіковано | Backend (авторитетно); Proxy може віддати за відсутнім/зламаним підписом на межі | Відсутній/невалідний підпис, невідомий `site-id`, timestamp поза вікном, повтор nonce (replay) |
| `403 Forbidden` | Токен відкликано/неактивний | Backend | Сайт знайдено, але токен `revoked`/`inactive` (FR-005) |
| `413 Payload Too Large` | Тіло завелике | Proxy | Розмір тіла/заголовків понад ліміт (дефолт 8 KiB) |
| `429 Too Many Requests` | Перевищено rate-limit | Proxy (1-й контур); Backend (2-й контур) | Понад per-`site-id` бюджет; заголовок `Retry-After` обов'язковий |
| `503 Service Unavailable` | Тимчасова недоступність | Proxy/Backend | Черга/сервіс тимчасово недоступні; безпечно для повтору плагіном |

Примітки семантики:
- `401` vs `403`: `401` — «не довів автентичність» (підпис/час/replay/невідомий id); `403` — «автентичність не потрібно доводити, доступ відкликано» (revoked). З міркувань нерозкриття (FR-032) допускається уніфікувати обидва як `401` без деталізації причини — точне рішення фіксується планом, але тіло в будь-якому разі узагальнене (2.4).
- Проксі віддає `429`/`413`/format-`400` **до** пересилання на бекенд (FR-026, FR-027).
- Плагін на `429`/`503` ПОВИНЕН відкласти повтор (шанувати `Retry-After`); на `401`/`403` — не ретраїти наосліп, а вимагати уваги вебмайстра (можлива ротація/відкликання секрету).

### 2.4 Узагальнена (нейтральна) відповідь про помилку

Усі помилки (і проксі, і бекенду) повертають узагальнене тіло, що **не** розкриває топологію, внутрішні адреси, стек, причину на рівні крипто-деталей чи існування інших сайтів (FR-019, FR-032, edge case «CRM повертає помилку плагіну»):

```jsonc
{
  "error": "request_rejected",          // стабільний машинний код зі скупого фіксованого набору
  "message": "Запит відхилено."          // нейтральний, локалізований, без деталей
}
```

Дозволений (приклад) скупий набір `error`-кодів: `bad_request`, `unauthorized`, `forbidden`, `payload_too_large`, `rate_limited`, `unavailable`. Заборонено: імена хостів/портів, внутрішні URL, назви сервісів БД/черги, вказівки «сайт X існує/не існує», значення секретів чи діагностику підпису.

## 3. CRM-internal operations (operator UI / admin API)

Ці операції — **не** plugin-facing. Вони доступні лише авторизованим операторам (адміністратор; менеджер — за майбутньою матрицею доступу) через захищену адмін-панель CRM (FR-022, A-5). Автентифікація — сесія/токен оператора (окремо від per-site HMAC; деталі — план). Усі три операції придатні до аудиту (хто/коли/тип/було→стало, FR-021).

> Шляхи наведено ілюстративно; контракт фіксує **форми та семантику**, не конкретний роутинг.

### 3.1 Реєстрація сайту

Реєструє сайт і видає пару облікових даних: **публічний `site-id` + одноразовий секрет** (FR-001, FR-002, FR-003, FR-004).

```
POST  /admin/sites
```

**Запит:**

```jsonc
{
  "name":   "<назва сайту>",       // обов'язково
  "domain": "<домен>"              // обов'язково; унікальний у межах мережі
}
```

**Успіх — `201 Created`:**

```jsonc
{
  "site": {
    "id":           "<внутрішній ID запису сайту>",
    "name":         "<назва>",
    "domain":       "<домен>",
    "status":       "pending",             // «очікує» (синій) — плагін ще не звітував
    "last_seen_at": null,
    "created_at":   "<ISO-8601>"
  },
  "credentials": {
    "site_id":        "<публічний site-ідентифікатор>",   // несекретний; можна показувати повторно
    "signing_secret": "<секрет — ПОКАЗУЄТЬСЯ ЄДИНИЙ РАЗ>", // FR-004: далі не читається у відкриту
    "sig_version":    "v1"
  }
}
```

Семантика:
- `signing_secret` повертається **рівно один раз** — у цій відповіді. Далі UI/API показує лише маскований плейсхолдер; за втрати — тільки перевипуск (FR-004, User Story 1 сценарій 3).
- Секрет: 256-біт CSPRNG, унікальний per-site, не виводиться з id/домену/іншого секрету (FR-003). На бекенді зберігається **encrypted-at-rest у відновлюваній формі** (envelope/KMS/app-level encrypter — ключ у env/секрет-сторі, не в репозиторії), бо HMAC потребує оригінал секрету для переобчислення (це шифрування, **не** односторонній хеш).
- `site_id` — публічний, opaque, high-entropy; ДОЗВОЛЕНО показувати повторно (FR-004).

**Помилка — дублікат домену — `409 Conflict`** (FR-006, User Story 1 сценарій 2):

```jsonc
{ "error": "domain_already_registered", "message": "Сайт із таким доменом уже існує." }
```

Дубль не створюється; відповідь повідомляє про існуючий сайт **лише авторизованому оператору** (це internal-контур, FR-019 стосується plugin-facing відповідей).

Інші коди: `400` (валідація name/domain), `401/403` (оператор не авторизований, FR-022).

### 3.2 Відкликання та перевипуск секрету

Відкликає поточний секрет і видає новий; старий перестає прийматися **майже одразу** (FR-005, User Story 4, SC-006). `site-id` **не змінюється** (A-4).

**Відкликання:**

```
POST  /admin/sites/{id}/credentials:revoke
```

Успіх — `200 OK`:

```jsonc
{
  "site_id": "<той самий публічний ідентифікатор>",
  "token_state": "revoked",
  "revoked_at": "<ISO-8601>"
}
```

Після цього будь-який heartbeat зі старим секретом відхиляється бекендом (`401/403`) у межах вікна поширення (дефолт ≤1 хв, SC-006). Сайт із часом стає `offline` за відсутністю валідних звітів (User Story 4 сценарій 3).

**Перевипуск (reissue):**

```
POST  /admin/sites/{id}/credentials:reissue
```

Успіх — `201 Created` (форма `credentials` — як у 3.1, секрет знову **показ один раз**):

```jsonc
{
  "credentials": {
    "site_id":        "<той самий публічний site-ідентифікатор>",   // НЕ змінюється
    "signing_secret": "<НОВИЙ секрет — показ один раз>",
    "sig_version":    "v1"
  },
  "previous_token_state": "revoked"                                  // старий недійсний
}
```

Семантика:
- Генерується новий унікальний секрет; старий лишається недійсним (User Story 4 сценарій 2).
- Один активний набір облікових даних на сайт.
- (Опційно, рішення плану) короткочасне вікно перекриття двох секретів (accept old+new) під час ротації, щоб сайт не впав в офлайн, поки вебмайстер оновлює конфіг плагіна (впливає на SC-002/SC-003). Для MVP допустимо без нього.
- Конкурентні дії двох операторів над одним сайтом ПОВИННІ давати узгоджений результат без гонок (edge case): реалізується оптимістичним контролем версії/послідовності (деталь плану; контракт вимагає детермінованого підсумкового стану).

Інші коди: `401/403` (авторизація), `404` (сайт не знайдено — узагальнено для operator-контуру).

### 3.3 Список сайтів зі статусом і часом останнього оновлення

Повертає всі зареєстровані сайти з поточним статусом підключення та часом останнього оновлення; підтримує фільтр за статусом із лічильником «N із M» (FR-015, FR-016, FR-018, User Story 3, `design/docs/CRM/02-sites-list.md`).

```
GET  /admin/sites?status={status}&page={n}&per_page={m}
```

Query-параметри (усі опційні):

| Параметр | Значення |
|---|---|
| `status` | фільтр: `online` \| `pending` \| `offline` \| `inactive` (без фільтра — усі) |
| `page`, `per_page` | пагінація |

**Успіх — `200 OK`:**

```jsonc
{
  "counts": {                       // для лічильника «N із M» (FR-018)
    "total":    120,                // M — усього сайтів
    "filtered": 14,                 // N — відповідають активному фільтру
    "by_status": {
      "online": 96, "pending": 6, "offline": 14, "inactive": 4
    }
  },
  "sites": [
    {
      "id":           "<ID>",
      "name":         "<назва>",
      "domain":       "<домен>",
      "status":       "offline",         // enum легенди статусів (див. нижче)
      "last_seen_at": "<ISO-8601 | null>", // час останнього прийнятого оновлення (FR-012, A-7)
      "token_state":  "active"           // active | revoked
    }
    // ...
  ],
  "page": 1,
  "per_page": 50
}
```

**Легенда `status`** (єдина, з дизайну — FR-016, A-3; бурштиновий «резерв» до сайтів **не** застосовується):

| `status` | Семантика | Колір легенди |
|---|---|---|
| `online` | Активний — валідний звіт у межах вікна | зелений (активний) |
| `pending` | «Очікує» — зареєстровано, плагін ще не звітував | синій |
| `offline` | Валідний звіт не надходив довше за вікно офлайну (FR-014) | червоний (помилка) |
| `inactive` | Вимкнений вручну оператором | сірий (неактивний) |

Семантика:
- Статус — це **збережений** (матеріалізований) стан, а не обчислення «на льоту»: `offline` виставляється плановою set-based детекцією за `last_seen_at < now − offline_window` (CRM **не** пінгує сайти, FR-014). Перехід `online→offline` та назад — точки емісії аудит/changelog-подій.
- Оновлення списку **не** зобов'язане бути real-time; показується останній оброблений стан (FR-017).
- `last_seen_at` — час останнього **прийнятого** heartbeat (FR-012); у рядку списку показується як «час останнього оновлення» (A-7).

Інші коди: `401/403` (авторизація, FR-022).

### 3.4 Деактивація та реактивація сайту

Ручне вимкнення/увімкнення сайту оператором — робить статус `inactive` (сірий) досяжним
(FR-013, FR-016; латч `inactive` у `data-model.md` §2). Придатне до аудиту (події
`site_deactivated` / `site_reactivated`, FR-021).

```
POST  /admin/sites/{id}:deactivate
POST  /admin/sites/{id}:reactivate
```

- **Деактивація** — `200 OK`: `status → inactive`, `deactivated_at = now()`. Поки сайт `inactive`,
  автоматичні heartbeat-транзиції придушені; вхідні heartbeat ДОЗВОЛЕНО відхиляти як `403` (сайт
  вимкнено) — точна політика фіксується планом. Тіло: `{ "id", "status": "inactive",
  "deactivated_at": "<ISO-8601>" }`.
- **Реактивація** — `200 OK`: `status → pending` (чекає наступного валідного heartbeat),
  `deactivated_at = null`. Тіло: `{ "id", "status": "pending" }`.
- Інші коди: `401/403` (авторизація, FR-022), `404` (сайт не знайдено).

> **Обсяг**: спека не має окремої user story на деактивацію, але FR-013/FR-016 прямо включають
> статус `inactive` як відображуваний — отже операція його встановлення **імпліцитна**. Мінімальна
> деактивація/реактивація робить статус досяжним і узгоджує контракт із `data-model.md`
> (усунуто розбіжність verify-пасу).

## Додаток C — Свідомо винесено на етап плану

- Конкретний продукт проксі/ingress (reverse-proxy / API-gateway / CDN з edge-логікою) — продукт-агностично; обов'язкові критерії відбору: rate-limit за заголовком (site-id, не IP), несекретна edge-перевірка без доступу до секретів, TLS re-encrypt/mTLS до приватного origin, корельований аудит (FR-033), приховування топології (FR-023, FR-032).
- Точний формат публічного `site-id` і кодування секрету; envelope/KMS vs Laravel encrypter; опційний override секрету/endpoint через wp-config константу на боці плагіна.
- Уніфікація `401`/`403` для heartbeat з міркувань нерозкриття (FR-032) — точна політика.
- Вікно перекриття двох секретів під час ротації (SC-002/SC-003) — увімкнути чи ні для MVP.
- Технологія nonce-стору (спільний, атомарний; TTL=600 с) та черги (async-приймання, FR-010).
- Уточнення формулювання конституції «хешованими на боці CRM» → для HMAC це **шифрування з можливістю розшифрування**, не односторонній хеш.

# Розгортання проксі/ingress-шлюзу (продукт-агностично)

Проксі/ingress — **обов'язкова** єдина публічна точка приймання перед CRM-бекендом
(FR-023..FR-033, contract §1.5). Конкретний продукт не фіксується специфікацією.

## Критерії відбору продукту

Придатний reverse-proxy / API-gateway / CDN з edge-логікою (nginx, Caddy, Envoy, HAProxy,
Cloudflare/Fastly workers тощо), що вміє:

- **TLS-термінацію** на межі (TLS 1.2+/1.3, HSTS, авто-оновлення сертифіката);
- **rate-limit за заголовком** `X-DB-Site-Id` (НЕ за IP — плагіни живуть за спільними хостингами);
- **дешеву несекретну edge-перевірку** без доступу до секретів (наявність/формат `X-DB-*`,
  вікно timestamp, ліміти розміру тіла/заголовків);
- **приховування бекенду** — внутрішня адреса/топологія CRM недосяжна ззовні (FR-023, FR-032).

## Дві ділянки TLS (обов'язково — Конституція v2.0.1)

```
[Плагін] --TLS--> [Проксі/Ingress] --TLS або mTLS--> [CRM-бекенд (приватна мережа)]
```

- Ділянка плагін↔проксі: TLS.
- Ділянка проксі↔бекенд: **не** cleartext — re-encrypt TLS **або** mTLS; бекенд у приватній
  мережі без публічного IP (приймає лише від проксі: firewall/security-group або mTLS).

## Defense-in-depth (проксі НЕ довірена ланка)

Проксі робить лише дешевий несекретний відсів. **Авторитетна перевірка — на бекенді**
(`HeartbeatController`): переобчислення HMAC (constant-time), timestamp, nonce, стан токена.
Бекенд **не** довіряє жодному доданому проксі заголовку (FR-028). Проксі **не** тримає per-site
HMAC-секретів (FR-027/FR-029) — його компрометація не дає підробити валідний підпис (FR-031).

## Кореляція аудиту (FR-033, T057)

Для розслідувань зв'яжіть edge-лог проксі з аудит-логом бекенду через **спільний
correlation-id**: значення `X-DB-Nonce` (унікальне на запит) або окремий `X-DB-Request-Id`.
Бекенд зберігає його в `event_log_entries.metadata.request_id` (`EventLogger`), проксі логує ту
саму пару при відхиленні/`429`/WAF. Секрети/повні підписи в логи НЕ пишуться.

## Приклад edge-перевірок на проксі (nginx-подібне, псевдо)

- вимагати присутність заголовків `X-DB-Site-Id`, `X-DB-Timestamp`, `X-DB-Nonce`, `X-DB-Signature`;
- `client_max_body_size 8k;` (ліміт тіла, FR-027);
- `limit_req_zone $http_x_db_site_id ...;` (rate-limit за site-id);
- проксіювати на приватний upstream бекенду поверх TLS/mTLS;
- на помилку віддавати **нейтральне** тіло (без топології).

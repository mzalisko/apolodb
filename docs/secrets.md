# Керування секретами

Конституція v2.0.1, секція «Вимоги безпеки». Два різні випадки зберігання:

## 1. Per-site секрети підпису (HMAC-ключі) — encrypted-at-rest

- Зберігаються **зашифрованими у відновлюваній формі** (Laravel encrypter, cast `encrypted`
  на `site_credentials.secret_encrypted`). HMAC-верифікація потребує **оригінал** секрету, тож
  односторонній хеш неможливий.
- Ключ шифрування — `APP_KEY` (env / секрет-сторі), **поза БД і поза репозиторієм**.
- Показується оператору **один раз** при видачі/перевипуску (FR-004); далі у відкриту недоступний.
- Індивідуальна ротація й миттєве відкликання per-site (`CredentialService::revoke/reissue`).
- **Продакшн**: `APP_KEY` подавати через сховище секретів (Vault/KMS/оточення платформи), не в
  образі й не в git. Розгляньте envelope-encryption/KMS для `secret_encrypted` як посилення.

## 2. Паролі операторів — one-way hash

- `users.password` — cast `hashed` (bcrypt/argon2). Відновлення не потрібне.

## Спільні правила

- Жоден секрет **не** потрапляє в репозиторій. `.gitignore` виключає `.env`, `*.key`, `*.pem`.
- Секрети **не** пишуться в аудит-лог (`EventLogger` фіксує лише факт видано/відкликано).
- На боці плагіна секрет — у WP Options API (`autoload=no`) або wp-config-константі
  `SD_SIGNING_SECRET`; ніколи не рендериться назад у форму (FR-024).

## Локальний dev / CI

- Dev-стек (docker-compose) використовує тестові креденшли `databridge/secret` — **лише для
  локального середовища**, не для продакшну.
- CI генерує `APP_KEY` на льоту (`php artisan key:generate`), БД — ефемерний контейнер.
